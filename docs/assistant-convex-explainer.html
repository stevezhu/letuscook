assistant-convex

A Convex-powered AI knowledge management backend that captures ideas, auto-organizes them into a knowledge graph, and generates rich documents — all driven by LLM intelligence.

What It Does

This is the backend brain of a personal AI assistant. Users feed it raw "captures" — text snippets, links, or tasks — and the system automatically:

  • Embeds each capture with Gemini vector embeddings (768-dim)
  • Finds semantically similar existing nodes via vector search
  • Generates a title and identifies organizing concepts via LLM
  • Builds a knowledge graph of linked nodes, edges, and virtual category nodes
  • Lets users review, accept, or reject AI suggestions before publishing
  • Generates rich markdown documents from node activity

Key Design Patterns

Scheduled Processing Pipeline

Mutations schedule actions via ctx.scheduler.runAfter(), enabling long-running AI work without blocking. Each phase hands off to the next asynchronously.

Draft → Publish Workflow

AI creates unpublished (draft) nodes and suggested edges. Users accept or reject before anything goes live. Rejection cleanly deletes all draft artifacts.

Vector + LLM Hybrid Search

768-dim Gemini embeddings power vector similarity (threshold 0.7), while LLMs extract organizing concepts and verify relationships — best of both worlds.

Multi-Model Fallback Chain

Primary: Gemini 2.5-flash → Fallback: OpenRouter → Tertiary: Gemini 3-flash. Retry logic with exponential backoff ensures resilience.

Soft Deletes Everywhere

All entities use archivedAt timestamps instead of hard deletes. Nodes cascade-archive their edges. Everything is restorable.

Explicit Mention Parsing

Users can link captures to existing nodes with @[Title](node:id) syntax, parsed at creation time and converted to suggested edges.

User Review Phase

Accept Suggestion

Publishes the draft node and all connected edges. Marks suggestion as accepted. The node enters the knowledge base.

Reject Suggestion

Deletes the draft node and all associated edges. Marks suggestion as rejected. Capture moves to needs_manual.

Organize Manually

User creates the node themselves with a custom title/content, linking explicit mentions. Bypasses AI entirely.

AI Operations Detail

What happens inside embedAndClassify

  • Link captures: Fetch HTML → extract OG meta, title, description, favicon, content snippet (500 chars)
  • Embedding: Send content to Gemini embedding model → 768-dim vector
  • Vector search: Query published nodes for cosine similarity ≥ 0.7 → top similar nodes
  • Title generation: LLM generates ≤80 char title with context of capture type + similar nodes (3-model fallback)
  • Organizing concepts: LLM identifies 1–3 category concepts → search or create virtual nodes → edge suggestions with confidence

Knowledge Graph Model

Nodes are the atoms of knowledge. Edges encode typed, directed relationships between them. Virtual nodes act as organizing categories that the AI creates automatically.

Edge Types

  • explicit — User-created manual links
  • suggested — AI-proposed from vector similarity
  • categorized_as — AI-assigned to virtual category nodes
  • related — Semantic similarity connections
  • reference — Direct content references

Node Types

  • regular — Concrete knowledge items created from captures (articles, tasks, ideas)
  • virtual — Organizing concepts created by AI (categories, themes). Not tied to a specific capture.

Edges with confidence > 0.8 are auto-verified. Hub nodes = high edge count.

The AI Engine — How It Really Works

Four source files in convex/ai/ power all intelligence: embedding.ts (vectors + titles), nodeLinker.ts (concept extraction), linkFetcher.ts (URL enrichment), and clustering.ts (k-means). The orchestrator is the embedAndClassify action in captures.ts, which chains them all together inside a single Convex action.

Multi-Model Fallback Chain

Every LLM call (title generation, concept identification, document generation) uses the same 3-model cascade. If a model fails or returns empty text, the system falls through to the next.

Primary google/gemini-2.5-flash
→
Backup openrouter/gemini-2.5-flash
→
Tertiary google/gemini-3-flash-preview

Retry Logic

Each model gets up to 3 attempts with exponential backoff: 1s → 2s → 4s (base delay 1000ms × 2attempt). If a model exhausts retries or returns empty output, the next model in the chain is tried. If all fail, a hardcoded fallback is used.

Step 1 — Vector Embedding

embedText(text) → number[768]

Sends raw text to Gemini Embedding 2 Preview with an explicit output dimensionality of 768. Returns a dense float64 vector stored directly on the node document. This vector powers all similarity search via Convex's built-in vector index.

// Model configuration const model = google.embeddingModel('gemini-embedding-2-preview') const options = { google: { outputDimensionality: 768 } } // For link captures, content is enriched first: // contentForEmbedding = ogTitle + ogDescription + contentSnippet // Falls back to rawContent if no metadata extracted
Step 2 — Vector Similarity Search

Finding Similar Nodes

After embedding, the system queries Convex's vector index to find up to 5 similar nodes, filtered to the current user and excluding unpublished/archived results. Only nodes with a cosine similarity score ≥ 0.7 survive filtering.

const results = await ctx.vectorSearch('nodes', 'by_embedding', { vector: embedding, limit: 5, filter: (q) => q.eq('ownerUserId', ownerUserId), }) // Post-filter: score >= 0.7, published, not archived // Returns: { id, title, content, score }[]
Step 3 — AI Title Generation

generateTitle(rawContent, captureType, similarNodes) → string

An LLM generates a concise ≤80-character title for the capture. Similar nodes are provided as context so titles stay consistent with the existing knowledge base.

System Prompt
Generate a concise, descriptive title (max 80 chars) for the following content. Return ONLY the title text, nothing else.
User Prompt (constructed dynamically)
Content type: ${captureType} Content: ${rawContent} Related existing nodes for context: ← only if similar nodes exist - "${node.title}": ${node.content.slice(0, 100)} - "${node.title}": ${node.content.slice(0, 100)}

Parameters: maxOutputTokens: 100. Final fallback: if all models fail, truncates rawContent.slice(0, 80) as the title.

Step 0 — Link Metadata Enrichment

For link type captures, this runs before embedding — the extracted metadata becomes the content that gets embedded.

fetchLinkMetadata(url) → LinkMetadataResult

Fetches the HTML page with a 5-second timeout and a custom User-Agent, then extracts metadata via regex patterns.

What Gets Extracted

og:title → title (fallback: <title> tag) og:description → description (fallback: meta description) og:image → ogImageUrl canonical → canonicalUrl favicon → link[rel*=icon] or /favicon.ico snippet → first 500 chars of body text

Content Snippet Pipeline

1. Remove <script> and <style> blocks 2. Strip all HTML tags 3. Decode entities: &amp; &lt; &gt; &quot; &#39; &nbsp; 4. Collapse whitespace → single space 5. Trim and slice to 500 chars

Fetch status: success if title OR description found · partial if neither · failed on HTTP error or timeout.

User-Agent: Mozilla/5.0 (compatible; LetUsCookBot/1.0; +https://letuscook.app)

Step 4 — Organizing Concept Identification

identifyOrganizingNodes(ctx, args) → NodeLinkSuggestion[]

An LLM reads the capture content and suggests 1–3 organizing concepts (category labels). Each concept is searched against existing nodes — if no match is found, a new virtual node is created automatically.

System Prompt
You are a knowledge organization assistant. Identify high-level organizing concepts/categories for content. Always respond with valid JSON only.
User Prompt
Given this ${captureType} content titled "${contentTitle}", suggest 1-3 organizing concepts or categories it belongs to. Each concept should be a short noun phrase (2-4 words) that could serve as a category label. Existing related nodes for reference: ← if similar nodes exist - "${similarNode.title}" Content: ${rawContent.slice(0, 500)} Return a JSON array only, no markdown, no explanation. Example format: [{"concept":"Machine Learning","confidence":0.9},{"concept":"Research Papers","confidence":0.75}]

Parameters: maxOutputTokens: 300

JSON Extraction & Fallback Parsing

The system tries JSON.parse() first. If that fails (markdown fences, extra text), it falls back to a regex extractor that handles flexible key quoting:

// Regex pattern for extracting concept/confidence pairs const pattern = /["']?concept["']?\s*:\s*["']([^"']+)["']\s*,\s*["']?confidence["']?\s*:\s*([\d.]+)/gi // Handles: "concept": "X", 'concept': 'X', concept: "X" // Confidence clamped to [0, 1]: Math.max(0, Math.min(1, val))

Concept → Node Resolution Flow

1 LLM returns concepts — e.g. [{concept: "Machine Learning", confidence: 0.9}]
↓
2 Search existing nodes — findNodesByTitle does case-insensitive substring match on all non-archived nodes owned by the user
↓
3 Match found? Use the existing node's ID · No match? Create a new virtual node (nodeKind: 'virtual', empty content, title = concept string)
↓
4 Return suggestions — each with { nodeId, edgeType: 'categorized_as', confidence, isNew }
Step 5 — Saving Results (saveEmbeddingResult)

What Gets Created

A single mutation that atomically creates all artifacts from the AI pipeline:

1. Draft Node

{ title: "AI-generated title", content: rawContent, searchText: title + "\n\n" + rawContent + (enriched if different), embedding: float64[768], ownerUserId, sourceCaptureId, // No publishedAt = draft (unpublished) }

2. Pending Suggestion

{ captureId, suggestorUserId: agentUser._id, suggestedNodeId: draftNode._id, status: "pending", }

3. Edges (three categories)

// Similar node edges edgeType: "suggested" source: "processor" verified: false confidence: score // 0.7–1.0
// Explicit mention edges edgeType: "suggested" source: "processor" verified: false // from @[Title](node:id)
// Organizing concept edges edgeType: "categorized_as" source: "processor" verified: confidence > 0.8 confidence: 0.0–1.0

Organizing edges with confidence > 0.8 are auto-verified — they skip the review step. Duplicate organizing edges are checked before insertion.

Bonus — Document Generation

generateDocumentAction(nodeId, version)

Generates a rich markdown document for any published node by gathering all its incoming edge activity, then sending it to the LLM.

Context Gathering

For each incoming edge to the target node: resolve the source node, its source capture, and any link metadata. This builds a full activity feed of everything connected to this knowledge node.

System Prompt
You are a knowledge assistant. Generate a comprehensive, well-structured markdown document summarizing the given knowledge node and its related content. The document should: - Start with a brief introduction/summary - Organize the related content into logical sections - Use markdown formatting with headers, bullet points where appropriate - Be informative and useful as a reference document - The first line should be a top-level heading with the document title Return ONLY the markdown content, no preamble or metadata.
User Prompt Structure
# Node: ${node.title} ## Node Description ${node.content} ## Related Content ### ${linkMetadata.title ?? fromNode.title} ${description} ${sourceContent.slice(0, 500)} ${snippet.slice(0, 500)} ...repeated for each connected edge/node...

The title is extracted from the first # heading in the generated markdown. Documents are versioned and track whether the user has manually edited them (isEdited flag).

Fallback content: # ${node.title}\n\nDocument generation failed. Please try again.

Utility — K-Means Clustering

kMeans(vectors, k) → Cluster[]

A full k-means++ implementation used for semantic grouping of nodes. Uses Euclidean distance over the 768-dimensional embedding space.

Initialization (k-means++)

1. Pick first centroid randomly 2. For each remaining centroid: // Weight each vector by squared distance // to nearest existing centroid d² = min(dist(v, c)²) for all centroids c // Probabilistic selection proportional to d² pick next centroid from weighted distribution 3. Result: k well-spread initial centroids

Iteration Loop

const MAX_ITER = 50 for each iteration: // Assign: each vector → nearest centroid assignments[i] = argmin(euclidean(v, c)) // Converge: if all assignments unchanged → break // Update: recalculate centroids as mean centroid[j] = mean(vectors in cluster j) // Output: clusters with members + distances // Empty clusters filtered out

Distance function: √(Σ(aᵢ - bᵢ)²) — standard Euclidean distance across all 768 dimensions.

Error Handling

The entire embedAndClassify action is wrapped in try/catch. On any unrecoverable error, it calls setCaptureFailed which sets the capture state to 'failed'. Users can retry failed captures via the retryProcessing mutation, which resets state to 'processing' and re-schedules the pipeline from the top.
